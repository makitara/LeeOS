# LeeOS Plugin Vibe

## 适用场景
- 新增/修改业务功能插件（UI/可视化/工具）。

## 沟通与对齐（必须）
- 开工前先问清楚：插件要解决什么问题？展示什么？数据从哪来？验收标准是什么？
- 如果需要 Host 新 API，先列“最小 API 清单”，不要直接改 Host。
- **严格按用户指令范围执行**：用户没要求的功能、路线、重构、延伸建议，一律不主动加入。
- **未经明确同意，不得添加兼容层/回退策略**：包括但不限于“文本映射兼容”“旧字段兼容”“自动兜底行为”等。
- **禁止添油加醋**：输出时不附加“下一步建议/可选优化/我建议你…”等内容，除非用户明确要求。
- **交付只陈述事实**：仅写已完成修改、验证方式、已知风险；不做主观产品方向引导。

## 不做的事
- 不修改 Host：禁止改 `Code/electron/**`、`Code/src/**`、`Code/vite.config.ts`、`Code/package.json`。
- 需要 Host 新能力时：列出最小 API 需求，并交给 `leeos-core-architect`。
- 不在用户未授权时更改交互范式、信息架构、视觉语言。
- 不在提示词中擅自加入“后续开发计划”段落，除非用户明确要求。

## 工作流程（简版）
1. 先写 `manifest.json`（`id` 必须唯一且与目录名一致）。
2. 入口默认 `index.html`（MVP 为静态插件）。
3. 数据通过 `LeeOS.fs.*` 写入插件专属数据目录。

## 关键交互硬约束（必须）
- **破坏性操作（删除/清空/覆盖）禁止依赖原生 `window.alert/confirm/prompt` 作为唯一交互**。
- 破坏性操作必须使用插件内可见、可回看状态的二步确认（如 `Delete -> Confirm`），并在执行中禁用冲突按钮。
- 新增关键按钮后，必须完成一次“点击链路自测”：`显示 -> 点击 -> 状态变化 -> 数据落盘 -> 视图刷新`。
- 若运行在 sandbox iframe，优先选择不依赖宿主弹框能力的方案。

## 提交前最小自测清单（必须）
- 关键路径按钮逐个点击验证（尤其新增按钮）。
- 至少覆盖一次失败路径（如非法输入/权限不可用/文件损坏）。
- 构建通过：`npm run build --silent`（在 `Code/` 下）。

## 交付输出格式（便于汇总）
最终输出尽量包含：
- **变更点**：按文件列出改动
- **如何验证**：命令 + 预期结果
- **风险/待确认**：仍需用户确认的点

### 提示词生成约束（新增）
- 当用户要求“给另一个 plugin-vibe 的提示词”时：
  - 只包含用户明确提供的目标、范围、约束、当前基线信息。
  - 不包含“接下来建议做什么”的任务规划，除非用户要求。
  - 不补充未被要求的产品决策。
